Pinagraph

Overview

Pinagraph is a geolocation social media app that I designed and programmed in 2014; long before I discovered UX.


Project Type:
Personal Project

Role:
UX/UI Designer
Front-end & Back-end Developer

Tools:
Paper + Pen
Sketch
PHP
HTML
CSS
JavaScript
jQuery
SQL
Mapbox


Problem

Today, younger generations are moving away from Facebook and migrating to more mobile-centric social media platforms like Instagram and Snapchat. My objective was to identify reasons for why Facebook is losing this demographic and solving this problem.


Challenge

Since I came from an academic background with no exposure to design and programming, my greatest challenge was to learn these skills and building the platform myself while being employed full-time. My first step to overcome the design challenge was trying to better conceptualize the design for my idea. I accomplished this by researching published articles and academic papers. Afterwards I proceeded with creating several paper prototypes for my desktop and mobile design. Then to overcome the programming challenge, I studied PHP, HTML, CSS, JavaScript, jQuery, and SQL. By 2015, I started coding the desktop version. I spent nearly every night coding from 9am-2am. Through this process, I encountered countless challenges. Some of which included troubleshooting bugs, solving complex code functions such as converting longitude and latitude on a photo’s metadata into a form that could be interpreted accurately in code, integrating Mapbox API, and constant re-designs. In the end what I had was a functional social network on desktop. Users could sign-up, add friends, post photos, geotag photos, send messages, etc.


Design Goal

Since mobile-based social media was the growing trend, I investigated other ways in which a mobile-based feature could be the inspiration for my design. This led me to discover that geolocation technology as being a plausible solution. Geolocation is entirely centered around mobile devices and the geolocation captures this sense of travel that heavy mobile-based users also share and cherish.


Process Outline

My process began with the 'Discover' phase by performing research. Afterwards I proceeded with 'Defining' my ideal user by performing interviews and created a user persona. I then continued with 'Designing' by creating paper prototypes for mobile and desktop. Finally, I 'Delivered' the platform by programming the entire database, back-end, and front-end for the desktop version.

Research

From my research, it was found that younger generations (specifically gen Z) are moving away from social media giant Facebook and looking to mobile-based social media like Instagram and Snapchat. The cause of such a movement was that gen z valued more privacy, more intimate circle of friends, prefer to interface with contents that were more relevant to them, and preserving their individuality.

Facebook on the other hand focused on expanding your network as much as possible and bombarded the user’s news feeds with viral videos, news, fake news, and advertisements. However, in the early days Facebook exploded in popularity because it was an exclusive social media for college students with a .edu email. News feeds were almost all contents that friends made;which are far more relevant to the users. This was one of the major formulas that replaced Myspace.

Desktop Application

The desktop application is built on PHP, HTML, CSS, Javascript, Jquery, and used MySQL for the database. For the geolocation application, I used the Mapbox’s API. For security, I hashed everything from passwords, to user IDs, and photos.



Index Page: Signup

For the sign-up, the code (exhibit 1) starts with executing the header.php. The code then ran a session function to check to see if the user was logged in. If the user was logged in, they would be forwarded to the home page; otherwise they would be prompted to sign in or to create an account.

For the signup, I coded the form using html (exhibit 2). After clicking the “Sign Up”, all of the user’s inputs get sent to the php’s post function (exhibit 1 row 10). I then programmed several lines of php codes that did a sanitary check to ensure that the user was not injecting malicious codes and to also check for errors. If everything was okay, all the inputs the user entered will be converted into a variable and then sent to the SQL database for registering the user.

Index Page: Login

For the login, a created a simple HTML form (exhibit 3) that allowed the user to input their email, password, and click the “Login” button. Once the “Login” button was clicked, the PHP script does a sanitary check to prevent any malicious code and then executes a query to the SQL database to see if there was a match between the email and password (exhibit 4 row 101). Home Page

Once the user was logged in, my home page started with declaring variables for the user’s friends. Then I ran a script to check to see if the user had any friends in the SQL database. For the user’s friend list, I created an array called “friend_array”. If the script found the array, it would then execute a lengthy if statement that separates the “friend_array” into variables and then give it a key (exhibit 5 row 34). Then for each friend that the script extracts from the database, I ran a foreach function that also retrieved the friend’s latest post to provide the news feed (exhibit 6 row 95).

Profile Page

For the profile page, I started off very similar to the home page by declaring all the variables at the top and then running a lengthy if statement to gather all the user’s friend list and posts. Afterwards, I have a JavaScript code that allows me to connect to the custom map that I created on Mapbox (exhibit 7 row 191).




Exhibit 7



Album Page

The album page also starts off similar to the home and profile page by declaring variables and then running a long if and foreach statement to pull all of the user’s photos and photo albums. Afterwards I added in a tool for users to upload their photos and then plot them on the map.

For the photo upload code, I started off with checking to see if the file type being uploaded met the file type and size restriction (exhibit 8 row 211). Afterwards, I renamed the image using a hash, sent the photo into the directory, and then recorded the directory of the photo into the SQL database. The reason each photo was renamed and hashed was to enforce user privacy by preventing people from guessing where these image files were.

To extract the geolocation from the photo, I first ran an “exif_read_data” script to read the metadata file (exhibit 9 row 236). If the metadata was found, I then ran a lengthy if statement to convert the information into coordinates that Mapbox could understand. Finally, for plotting the photo on to the map, I used JSON that Mapbox provided and used their markers (exhibit 10 row 123).
